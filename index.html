01 — Fundamentals

What Is a Multi-Agent System?

A multi-agent system is an architecture where multiple specialized AI agents — each with their own instructions, tools, and context — cooperate to accomplish tasks that would be too complex or too broad for a single agent. Instead of one monolithic prompt that tries to do everything, you decompose work into focused roles.

Every agent is fundamentally an LLM + instructions + tools. The magic happens in how they communicate: through handoffs (transferring control), tool calls (delegating sub-tasks), and shared context (conversation history and state).

🤖

Agent

An LLM configured with specific instructions, a set of tools, and optional guardrails. The atomic unit of work.

🔧

Tool

A function the agent can invoke — API calls, database queries, file operations, shell commands, or even calling another agent.

🤝

Handoff

A control transfer from one agent to another. The current agent decides another agent is better suited, and passes along the conversation state.

🛡️

Guardrail

Safety checks that validate inputs and outputs — running in parallel with agent execution. Catches policy violations before they cause harm.

📊

Tracing

Built-in observability: every LLM call, tool invocation, handoff, and guardrail check is recorded for debugging and optimization.

💾

Session

Persistent conversation state that survives across turns. Agents reload context from sessions, maintaining continuity.

02 — Core Mechanism

The Agent Loop

At the heart of every agent — single or multi — is the agent loop. This is the iterative cycle where the model reasons, decides on actions, executes tools, observes results, and either continues acting or produces a final output. This pattern is sometimes called the ReAct loop (Reason + Act).

Fig 1 — The fundamental agent loop: input → context → inference → tool execution → loop or reply

The loop terminates when the LLM produces text output without any tool calls — that's the final output. If the model calls a tool, the result is appended to context and the model is called again. If it performs a handoff, the current agent is replaced and the loop restarts with the new agent's instructions.

Key Insight

This loop is universal. OpenClaw, the OpenAI Agents SDK, Claude Code, LangGraph — every serious agent framework runs some version of it. The differences lie in what wraps the loop: session management, memory persistence, routing, security, and skill injection.

The Loop Step-by-Step

1
Message Received
A user message arrives via a channel (chat UI, API, WhatsApp, Slack, etc.). The orchestration layer identifies the correct session and queues the message.
2
Context Assembly
The system prompt is built dynamically by merging: base instructions, persona files (SOUL.md), active skills, tool schemas, session history, and memory. Token limits are enforced — if history exceeds the context window, compaction or summarization occurs.
3
Input Guardrails
Before (or in parallel with) model inference, input guardrails validate the user's message. These can be LLM-based classifiers, regex rules, or moderation API calls. If a guardrail trips, execution halts immediately.
4
LLM Inference
The assembled prompt is sent to the model. The model streams its response, which may contain plain text, one or more tool calls, or a handoff instruction.
5
Decision Gate
If the response contains no tool calls and matches the expected output type → the loop ends and the text is the final output. If it contains tool calls → proceed to execution. If it's a handoff → swap the active agent and restart.
6
Tool Execution
Each tool call passes through a policy chain (global → provider → agent → session → sandbox). If approved, the tool function runs and returns structured results. Results are appended to the conversation as tool-result messages.
7
Loop Iteration
The updated context (now including tool results) is sent back to the model. The loop repeats from step 4. A max_turns limit prevents infinite loops.
8
Output Guardrails
Once the final output is produced, output guardrails validate it before delivery. These check for policy compliance, format correctness, and safety.
9
Response Delivery
The final response is streamed back through the channel adapter to the user's platform. Streaming chunks are managed per platform's requirements.
10
Persistence
The full conversation turn — user message, tool calls, tool results, and assistant reply — is written to the session transcript (typically JSONL). Memory managers may run post-compaction to extract durable knowledge.
03 — Scaling Up

From Single Agent to Multi-Agent

A single agent with the right tools can handle a remarkable amount of work. The recommendation from OpenAI's own guidance is to maximize a single agent's capabilities first, then split when the toolset and branching logic get unwieldy. Teams typically move to multi-agent setups for four reasons:

🔀

Routing Complexity

One agent does triage and routing. Different types of requests need fundamentally different instructions and tools.

📚

Knowledge Separation

One agent handles retrieval and grounding. Specialized knowledge domains are too large to fit in a single prompt.

⚡

Action Isolation

One agent performs write operations. High-risk actions (sending emails, modifying records) need separate permissions and approval gates.

✅

Review & Policy

One agent reviews, checks policy, or asks for approval. Quality control is best handled by a separate "brain" with different instructions.

04 — Architecture

Orchestration Patterns

Multi-agent systems can be modeled as graphs, with agents as nodes and communication as edges. The two primary patterns are the manager (hierarchical) pattern and the decentralized (handoff) pattern. A third common pattern involves agents-as-tools, where one agent calls another like a function.

Pattern 1: Manager (Hierarchical)

A central manager agent receives all user input, delegates tasks to specialist agents via tool calls, and synthesizes their outputs into a unified response. The manager maintains control throughout — specialists never talk to the user directly.

Fig 3 — Manager pattern: specialists are called as tools and return results to the orchestrator

Pattern 2: Decentralized (Handoffs)

In the handoff pattern, agents are peers. When one agent decides another is better suited for the current task, it performs a handoff — a one-way transfer of control that passes the full conversation state to the next agent. There is no central orchestrator; agents form a network of possible transitions.

Fig 4 — Handoff pattern: agents transfer control peer-to-peer based on intent classification

Pattern 3: Chaining & Parallel

Two additional patterns complete the picture. In chaining, the output of one agent becomes the input of the next — like a pipeline (research → outline → write → critique → improve). In parallel execution, independent tasks are dispatched simultaneously using primitives like asyncio.gather(), then results are collected and synthesized.

05 — Control Transfer

Handoffs in Detail

A handoff is the mechanism by which one agent transfers full control of a conversation to another agent. In the OpenAI Agents SDK, handoffs are represented as tools to the LLM — if there's a handoff to a "Refund Agent", the model sees a tool named transfer_to_refund_agent.

How a Handoff Works Internally

1
Agent Decides
The current agent's LLM decides that another agent is better suited. It calls the handoff tool (e.g., transfer_to_refund_agent), optionally providing structured input like an escalation reason.
2
Callback Fires
If an on_handoff callback is registered, it executes now. This is useful for logging, data prefetching, or notifying external systems that a transfer is happening.
3
Input Filter Applies
An optional input_filter transforms the conversation history before the new agent sees it. Common filters remove all tool-call noise, summarize history, or strip sensitive information.
4
Agent Swap
The runner updates the current agent reference. The new agent's instructions, tools, model, and guardrails replace the previous agent's. The conversation continues with the new agent's identity.
5
Loop Restarts
The agent loop resumes from the LLM inference step with the new agent. The new agent now "owns" the conversation until it produces a final output or performs another handoff.

Code Example — OpenAI Agents SDK

Python — OpenAI Agents SDK
from agents import Agent, Runner, handoff

billing_agent = Agent(
    name="Billing agent",
    instructions="You handle billing questions and payment issues.",
)

refund_agent = Agent(
    name="Refund agent",
    instructions="You process refunds and returns.",
)

triage_agent = Agent(
    name="Triage agent",
    instructions="Route to the appropriate agent based on the request.",
    handoffs=[billing_agent, handoff(refund_agent)],
)

# The triage agent can hand off to either specialist.
# Handoffs appear as tools: transfer_to_billing_agent, transfer_to_refund_agent
result = await Runner.run(triage_agent, "I want a refund for my last order.")
print(result.final_output)
# → The refund agent processes the request
06 — Safety

Guardrails

Guardrails are configurable safety checks that validate inputs and outputs at every stage. Think of them as a layered defense mechanism — a single guardrail won't catch everything, but multiple specialized guardrails together create a resilient system.

Fig 5 — Guardrails operate at both input and output boundaries, with tripwire-based halting

Input guardrails can run in parallel with agent execution (for low latency) or before the agent starts (to prevent token consumption on invalid inputs). Output guardrails always run after the agent completes. Tool guardrails wrap individual tools and validate arguments before and after each tool call.

Tripwires

When a guardrail detects a violation, it triggers a tripwire which raises an exception (InputGuardrailTripwireTriggered or OutputGuardrailTripwireTriggered). This immediately halts the agent run, preventing any further tool calls or output delivery.

07 — Case Study

OpenClaw: Multi-Agent in the Real World

OpenClaw is an open-source autonomous AI agent framework created by Peter Steinberger that surpassed 200k+ GitHub stars in early 2026. It's a production-grade example of multi-agent architecture — not just the agent loop, but everything around it: routing, sessions, memory, skills, and multi-channel delivery.

Architecture Overview

OpenClaw's architecture breaks into four layers. The key insight is that the core agent loop (ReAct cycle) is delegated to the Pi Agent framework — OpenClaw itself focuses on the gateway, orchestration, integration, and intelligence layers that make the agent loop useful in production.

Fig 6 — OpenClaw's four-layer architecture: Channels → Gateway → Agent Runtime → Intelligence

OpenClaw's Multi-Agent Routing

In OpenClaw, each agent is a fully isolated persona with its own workspace (SOUL.md, AGENTS.md), state directory, session store, authentication profiles, and skills. The Gateway routes inbound messages to agents using bindings — deterministic rules that match on channel, account, peer identity, guild, or team.

This means you can run multiple agents on a single Gateway server: a "work" agent on one Telegram bot, a "personal" agent on WhatsApp, a "coding" agent on Discord — all isolated, all with different personalities, tools, and models.

Execution Sequence (per message)

1
Channel Adapter Receives
Platform-specific adapter (Baileys for WhatsApp, grammY for Telegram, etc.) normalizes the inbound message into a unified format, extracting attachments and metadata.
2
Gateway Routes to Session
The session manager resolves sender identity, matches bindings to determine agentId, and assigns a session key. The lane queue enforces serial execution per session.
3
System Prompt Assembled
Context is built from: IDENTITY.md → SOUL.md → TOOLS.md → active skills (selectively injected per turn) → memory files → conversation history. Token limits enforced.
4
Pi Agent Loop Executes
The Pi Agent core runs the ReAct loop: LLM inference → tool calls (with 5-level policy chain validation) → results fed back → loop until no tool calls remain.
5
Response Streamed Back
Text deltas are buffered and streamed to the channel adapter, which translates back into platform-specific format. Tool events are also emitted for debugging.
6
Transcript Persisted
The complete turn is written to a JSONL session transcript. The memory manager may run compaction to extract durable knowledge into MEMORY.md.
Key Design Principle

OpenClaw doesn't implement its own agent runtime — it delegates the "think and act" cycle to Pi Agent and focuses on everything else: channel normalization, session management, memory persistence, skill extensibility, and security enforcement. This separation reflects the insight that the hard problem in production agents isn't the loop itself, but the infrastructure around it.

08 — Framework

OpenAI Agents SDK

The OpenAI Agents SDK is a production-ready framework (successor to the experimental Swarm project) built around three primitives: Agents, Handoffs, and Guardrails. It's deliberately lightweight — you express workflow logic using familiar Python constructs rather than defining explicit graph structures.

Feature Description When to Use
Agents LLMs configured with instructions, tools, guardrails, model settings, and handoffs Always — the atomic building block
Handoffs Control transfer from one agent to another, with optional input filtering and callbacks When different requests need fundamentally different agents
Guardrails Input/output/tool validation with tripwire-based halting, runs parallel or blocking Always in production — defense in depth
Sessions Persistent memory layer for maintaining context within and across agent runs Multi-turn conversations, long-running agents
Tracing Built-in tracking of LLM calls, tool invocations, handoffs, guardrails — viewable in dashboard Always — essential for debugging and optimization
Function Tools Turn any Python function into a tool with automatic schema generation and Pydantic validation Extending agent capabilities with custom logic
MCP Servers Built-in Model Context Protocol integration — works the same as function tools Integrating with external services (GitHub, Slack, etc.)

The Runner Loop

When you call Runner.run(), this is what happens internally:

1
Call the LLM
Send the current agent's instructions, tools, and conversation input to the configured model.
2
Check Output
If the LLM returns a final_output (text with no tool calls) → the loop ends and returns the result.
3
Handle Handoff
If the LLM performs a handoff → update the current agent and input, and re-run the loop with the new agent.
4
Execute Tool Calls
If the LLM produces tool calls → run those tool functions, append the results, and re-run the loop.
5
Safety Check
If max_turns is exceeded → raise a MaxTurnsExceeded exception to prevent infinite loops.

Complete Multi-Agent Example

Python — Multi-Agent Customer Support
from agents import Agent, Runner, handoff, function_tool, input_guardrail
from agents import GuardrailFunctionOutput, RunContextWrapper

# ── Tool: look up an order ──
@function_tool
def lookup_order(order_id: str) -> str:
    """Look up order status by ID."""
    return f"Order {order_id}: Shipped, arriving March 1"

# ── Guardrail: block off-topic math homework ──
@input_guardrail
async def no_homework(ctx: RunContextWrapper, agent, input):
    # Use a cheap model to classify
    result = await Runner.run(classifier, input)
    return GuardrailFunctionOutput(
        tripwire_triggered="homework" in result.final_output
    )

# ── Specialist agents ──
orders_agent = Agent(
    name="Orders",
    instructions="Help customers track and manage their orders.",
    tools=[lookup_order],
)

refund_agent = Agent(
    name="Refunds",
    instructions="Process refund requests. Always confirm the order ID.",
)

# ── Triage agent with handoffs + guardrails ──
triage = Agent(
    name="Support Triage",
    instructions="Route customers to the right specialist.",
    handoffs=[orders_agent, handoff(refund_agent)],
    input_guardrails=[no_homework],
)

# ── Run ──
result = await Runner.run(triage, "Where is my order #12345?")
print(result.final_output)
# → "Your order #12345 has shipped and is arriving March 1!"
09 — Observability

Tracing & Debugging

In a multi-agent system, understanding what happened is as important as making it happen. Tracing captures a comprehensive record of every event during an agent run: LLM generations, tool calls, handoffs, guardrail checks, and timing data.

Fig 7 — Trace timeline showing a complete multi-agent request with handoff, tool call, and guardrail

In the OpenAI Agents SDK, tracing is automatic — every Runner.run() call is wrapped in a trace, and each agent execution, tool call, and handoff gets its own span. You can view traces in the OpenAI dashboard to debug exactly what happened and why.

In OpenClaw, every turn is written to a JSONL session transcript — a line-by-line audit of messages, tool calls, execution results, and timing. This provides full replay capability and serves as the basis for memory extraction and compaction.

10 — Deployment

Production Checklist

Moving from a prototype to production multi-agent system requires attention to reliability, security, and observability. Here's a distilled checklist drawn from both the OpenAI Agents SDK guidance and OpenClaw's production architecture.

✓
Start with a single agent
Maximize capabilities with one agent first. Only split when the toolset and branching logic make a single prompt unwieldy. Premature multi-agent adds complexity without benefit.
✓
Layer guardrails at every stage
Input guardrails to block invalid/malicious input. Tool guardrails to validate arguments. Output guardrails to ensure compliance. No single guardrail is sufficient — combine LLM classifiers, regex, and moderation APIs.
✓
Enforce serial execution per session
Prevent race conditions by serializing requests within a session. OpenClaw's lane queue system defaults to this. Parallelism should be opt-in for explicitly safe tasks.
✓
Set max_turns limits
Every agent run needs a maximum turn count to prevent infinite tool-calling loops. If exceeded, raise an exception and alert — don't let agents run forever.
✓
Implement human-in-the-loop gates
For high-risk actions (sending emails, making payments, modifying records), require explicit human approval. Use tool approval workflows to gate dangerous operations.
✓
Enable full tracing
Every LLM call, tool invocation, handoff, and guardrail check should be traced. This is non-negotiable for debugging production issues and optimizing performance.
✓
Use model-appropriate sizing
Not every task needs the most powerful model. Use frontier models for complex orchestration and cheaper/faster models for classification, simple retrieval, and guardrail checks.
✓
Manage context windows carefully
Monitor token counts. Implement compaction/summarization before hitting limits. OpenClaw uses a Context Window Guard that triggers summarization before the window overflows.
✓
Build evaluation pipelines
Use trace grading to assess end-to-end workflow performance. Run automated evals on representative inputs. Measure accuracy, latency, cost, and safety metrics.
✓
Iterate with real users
The path to production isn't all-or-nothing. Start small, validate with real users, and grow capabilities over time. Agents are fundamentally unpredictable — testing in the real world is essential.
